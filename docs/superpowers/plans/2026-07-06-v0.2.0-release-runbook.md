# v0.2.0 Release Runbook

Pre-verified on 2026-07-06 (see .superpowers/sdd/progress.md). Every step below is copy-paste; nothing else is required.

## 1. Integrate the release branch into main (linear history)

One-time gh fix first (the repo has two remotes, origin + private, so gh needs a default):

    gh repo set-default tinydantic/tinydantic

Verified 2026-07-06: `main` has NOT diverged — the release branch is a strict descendant (55 commits ahead), so main can simply fast-forward. Two linear-history options:

**Option A — fast-forward (preferred if branch protection lets you push to main):** no new commits, main ends up on the exact SHAs that CI tested and every commit keeps its signature/trailers byte-for-byte.

    git checkout main && git pull
    git merge --ff-only cdwilson/v0.2.0-m5-release
    git push origin main

**Option B — PR with "Rebase and merge" (if a PR is required):**

    gh pr create --base main --head cdwilson/v0.2.0-m5-release \
      --title "tinydantic v0.2.0: TinydanticModel, complete Table API, uv toolchain" \
      --fill-verbose

Review, wait for PR CI (14 jobs), then choose **Rebase and merge**. Since main hasn't diverged the content is identical, but GitHub still rewrites all 55 SHAs — commit MESSAGES (the feat!/BREAKING CHANGE data cz bump reads) are preserved, so the version math is unaffected. Caveats: GitHub does NOT sign rebase-merged commits (fails if branch protection also requires signed commits — use Option A then), and your local branches won't match main's SHAs afterward (cosmetic).

Do NOT squash: it collapses the conventional commits cz bump reads.

## 2. Confirm the docs-dev deploy on main

    gh run watch $(gh run list --workflow docs-dev.yaml --limit 1 \
      --json databaseId -q '.[0].databaseId') --exit-status

(mike >=2.2.0 detects the installed properdocs package and builds with it — same engine as the CI gate.)

## 3. Cut the release from main

    git checkout main && git pull
    uv run cz bump    # writes version 0.2.0 + creates tag v0.2.0
    git push origin main --follow-tags

## 4. Watch the publish pipeline

The tag triggers: version-guard -> build -> publish-to-pypi (+ GitHub Release with sigstore artifacts) and docs-release (mike deploys the v0.2.0 docs alias).

    gh run watch $(gh run list --workflow publish-python-package.yaml \
      --limit 1 --json databaseId -q '.[0].databaseId') --exit-status

## 5. Post-release sanity

    pip install tinydantic==0.2.0   # in a scratch venv

Check <https://tinydantic.dev> shows the new usage pages.

## Notes

- The publish version-guard has never fired before (first tagged release on the new pipeline) — step 4's watch is where it proves out. If it fails, the tag/pyproject mismatch message says exactly what's wrong; nothing publishes on failure.
- gh CLI was hitting keychain auth timeouts overnight; if it still does, re-auth with `gh auth login` or use the GitHub UI for 1-2.
